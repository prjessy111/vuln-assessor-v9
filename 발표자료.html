과제명 · AI 보안 취약점 진단 정합성 자동화 부서 · 제품전략팀 참여자 · 송훈근

왜 필요한가

고객사가 SecuMS 점검 결과의 정합성 확인을 요청하면, 지금은 판단 기준도 없이 담당자가 매번 수작업으로 확인 → 시간 과다. 이 과제는 3가지 점검 결과를 비교해 정합성 기준·방법을 확립하고, 고객 정합성 요청·신규 항목 점검에 활용한다.

1. Pain Point

  • 정합성 확인이 전부 수작업 — 고객 요청 시 담당자가 항목별로 일일이 확인, 시간 과다.
  • 정합성 기준의 부재 — 판정이 맞는지 판단할 표준·방법이 없어 매번 즉흥적.
  • 신규 항목 대응 지연 — 새 점검 항목 요청 시 그 정합성 확인에도 시간이 크게 소요.

2. 기존 업무 방식

SecuMS 진단→ 고객사 정합성 확인 요청→ 담당자 수작업 확인→ 시간 과다

정합성을 판단할 기준·자동화된 방법이 없는 것이 근본 문제.

3. AI 자동화 적용 — 핵심 골격

수집(raw)과 판정(AI)을 분리하고, 아래 3가지 결과를 비교해 정합성을 검증한다.

① 스크립트 AI 점검
스크립트 배포·수집 → AI 판정
② SecuMS raw AI 점검
SecuMS 원본 데이터 → AI 판정
③ SecuMS 자체 점검
정답지(기준)

세 결과를 항목별로 비교 → SecuMS 기준 90% 정합성을 1차 목표로 검증하고, 그 방법을 표준화한다.

4. Prompt / Context / Harness 표준

Prompt
2026 기준(확인방법·기준·조치법) + 항목별 값-의미 규칙 주입
Context
서버 OS · 수집 raw 증거 · 2026 금융보안원 OS 점검 기준(OS별 약 70항목) · SecuMS 정답지
Harness
AI 1차 + LLM 2차 2-pass · 자동 배포(WinRM/SSH) · 정합성 자동 측정 · CVE 진단 병행

판정 기준을 코드가 아닌 프롬프트·기준표(JSON)로 외부화 → 변경·고도화·재현이 쉬움.

5. 실제 동작 / 데모

  • 수집관리 — 다중 서버 선택 → OS별 표준 스크립트 자동 배포·진단
  • 진단 결과 — 항목별 취약/양호/정보/판정불가 + 사유·증거·조치
  • 정합성 리포트 — AI 판정 ↔ SecuMS 정답지 비교, 일치율 표시

7. 보안 · 검증 기준

  • 정답지 기반 검증 — 운영 스캐너(SecuMS) 결과 대비 항목별 일치율을 정량 측정.
  • 표준 준거 — 금융보안원 전자금융기반시설(2026) OS 점검 기준.
  • 수집/판정 분리 — 증거 없으면 억지 판정 대신 "판정불가"로(과탐 방지).
