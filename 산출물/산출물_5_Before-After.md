## H. 정합성 해석 · 개선안 · 향후 방향

### (1) 왜 정합성이 "단일 고정값"이 아닌가 — 구조적 이유
LLM 기반 자동 판정은 본질적으로 매번 똑같은 단일 정합성 값이 나오지 않는다. 네 가지 이유:
1. **측정 지표가 여러 층위** — N/A율(판정불가) / 2-way(우리 AI↔SecuMS 정답지) / 3-way(SecuMS-AI=Script-AI=정답지 **모두** 일치). 어느 지표를 보느냐에 따라 수치가 다르다(3-way가 가장 엄격). "다 맞췄는데 또 틀리네"의 주원인 = 더 엄격한 지표로 넘어갔기 때문.
2. **LLM 비결정성** — 동일 입력·temperature 0 이어도 경계(borderline) 항목은 실행마다 취약↔양호↔판정불가로 ±수 % 흔들린다. 룰 엔진과 달리 100% 재현은 불가.
3. **두 AI 소스의 독립성** — SecuMS raw DB 진단과 Script 수집 진단은 서로 다른 데이터로 독립 판정 → 각각 정답지엔 맞아도 서로 100% 일치하진 않는다.
4. **방어적 판정(의도된 불일치)** — 일부 항목은 우리 AI가 SecuMS보다 엄격(내부 서비스 하드닝 미설정, SUID 화이트리스트 거부 등). 미탐(false negative)을 줄이는 의도된 차이로, 무조건 SecuMS에 맞추면 오히려 보안 위험.

### (2) 개선안 (이번 적용 + 잔여 관리)
**적용 완료:**
- 수집 스크립트 AI친화화(`SERVICE_PRESENCE`/`COLLECTION_HINT`/`fOrElse`, 잘못된 수집 교정) → N/A 제거
- 과탐방지·부재양호·localhost 규칙 + 항목별 `SRV_HINTS` → 과탐/미탐 교정 (Script-AI 78.3%→88.3%)
- 3-way 리포트 상단에 **정합성(모수 N개 중 M개)** 상시 표시 → 수치로 모니터링

**잔여(관리 대상):**
- 경계 항목 LLM 변동 → **N회 다수결**(동일 항목 복수 판정 합의)로 변동 축소
- 방어적 판정 차이 → **엄격/표준 모드** 정책으로 명문화

### (3) 향후 방향성
1. **판정 안정화** — 경계 항목은 N회 다수결(복수 판정 합의)로 실행별 흔들림 최소화.
2. **신규 항목 자동 점검 AI Agent** — SecuMS 미커버 항목의 점검 절차(명령)를 자동 생성→수집→판정(Claude Skill 기반).
3. **수집 완전성 강화** — 권한 부족 항목(`/etc/shadow` 등) sudo 수집, 항목↔수집명령 매핑(crosswalk) 정기 검증.
4. **정합성 운영 체계화** — 정합성 기준선(SLA) 설정, 매 진단 정합성 회귀 모니터링, 방어적 판정 정책 문서화.
