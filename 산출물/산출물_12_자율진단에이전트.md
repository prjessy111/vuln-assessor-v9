# 산출물 12 — 자율 진단 에이전트 (Claude Skill 기반 자율 진단 루프)

> 신규 핵심 차별 기능. "진단 도구"를 넘어, **신규 점검 항목을 자연어로 정의하면
> 스크립트 생성 → 실행 → 판정·검증까지 사람 개입 최소로 자동 수행**하는 에이전트.
> 대상: Vuln Assessor v9 / 판정 LLM: **사내 LSAP(qwen3.5-122b-fast)** / 보고: Hermes 메신저 게이트웨이.

---

## 1. 한 줄 요약
**"공장(기존 진단 시스템)" 위에 "작업반장(에이전트)"을 얹었다.**
신규 종목이 들어와도 사람이 룰·스크립트를 일일이 짜지 않고, 자연어 설명만 주면
에이전트가 점검 스크립트와 판정을 만들고, 사람은 **두 개의 게이트(실행 승인 / 판정 확정)**만 검수한다.

## 2. 파이프라인 (5단계 + 사람 게이트 2개)
```
① 신규 항목 편입(자연어)
   → ② 점검 스크립트 자동 생성 (사내 LSAP)  [안전게이트: 위험명령 정적 차단]
   → 🚦 게이트1: 사람 실행 승인
   → ③ 대상 실행/수집 (Linux=SSH · Windows=WinRM, 또는 폐쇄망=결과 붙여넣기)
   → ④ raw 기반 자동 판정 (양호/취약/점검불가 + 근거 + 조치 + 신뢰도)
   → ⚖ 교차검증 → 3소스 합의(LSAP / 재검토 / SecuMS)
   → 🚦 게이트2: 불일치·애매 항목만 사람 최종 확정
   → 확정 항목 → 룰 YAML 편입 · 결과 요약 메신저 보고
```

## 3. 보안 설계 (사내 우선·외부 차단)
- **판정·생성 전부 사내 LSAP**(qwen, 무료·내부). raw 점검 데이터 외부 유출 없음.
- **외부 LLM(Claude)은 기본 차단** — `AGENT_ALLOW_EXTERNAL=true` 명시 없이는 코드 레벨에서 거부.
- **메신저 보고는 "판정 요약"만** 전송(raw 미포함). 폐쇄망에서는 결과 붙여넣기 방식 지원.
- **사람 확인 게이트 2개를 코드로 강제** — 승인 전 실행 불가, blocked 스크립트 승인 불가, 애매 판정은 확정 전 통과 불가.

## 4. 실측 결과 (사내 LSAP 실연결)
| 시나리오 | 결과 |
|---|---|
| 스크립트 자동 생성 (SSH root 점검) | LSAP가 `/etc/ssh/sshd_config` 조회 **읽기전용 sh** 생성, 안전도 safe |
| **jessy207 실제 SSH 실행** (192.168.159.207) | `permitrootlogin yes` 수집 → **취약(신뢰도 1.00)** + 조치 자동 작성 |

| **메신저 보고** | 판정 요약 텔레그램 전송 성공(요약만, raw 비전송) |
| 회귀 테스트 | 자동 테스트 20케이스 통과(흐름·게이트·안전검사·합의·보안차단·룰출력) |

## 5. 구성 모듈 (`src/agent/`)
| 모듈 | 역할 |
|---|---|
| `itemRegistry` | 자연어 항목 편입 + 상태머신(draft→…→confirmed) |
| `scriptGenerator` | 항목 설명 → sh/PowerShell 읽기전용 점검 스크립트 (LSAP) |
| `safetyGate` | 생성 스크립트 위험명령 정적 차단(blocked/warning/safe) |
| `scriptRunner` | 승인 스크립트 원격 실행(SSH/WinRM), 자격증명=servers.csv |
| `autoJudge` | raw → 양호/취약/점검불가 + 근거·조치·신뢰도, 저신뢰·불일치 검토 플래그 |
| `crossVerify` | 사내 재검토 → LSAP/재검토/SecuMS 3소스 합의(§4-4) |
| `ruleExport` | 확정 항목 → v2 룰 YAML(룰셋 편입) |
| `hermesNotify` | 결과 요약 메신저(Hermes) 보고 |
| `pipeline` | 단계 연결 + 사람 게이트 강제 |

## 6. 의의 / 확장
- **사람의 일이 "전수 작성"에서 "예외 검수"로 축소** — 합의된 항목은 통과, 엇갈린 것만 사람이 본다(검증 시스템의 핵심 가치).
- **확장 비용 절감** — 신규 점검 항목(같은 OS 내 신규 CVE/가이드/고객 기준)을 자연어로만 추가. <br>※ **현재 대상은 OS(Linux/Windows)뿐.** OS에서 쉽게 되는 건 기존 점검·**정합성을 많이 맞춰뒀고 환경(서버·자격증명·SSH/WinRM·LLM)도 세팅돼 있어서**다. DB/WAS·네트워크 등으로 확장하려면 **그 대상의 점검·정합성 토대부터 새로** 갖춰야 하므로 별도 과제(§7는 그 아이디어).
- **하이브리드 검증** — 전수는 무료 사내 LSAP, 핵심·애매 항목만 선택적 2차 검증으로 정확도와 비용을 동시에.

## 7. 에이전트로 구현 가능한 아이디어
> 핵심 메커니즘(자연어 항목 → 스크립트 생성 → 실행 → 판정 → 보고, 사람 게이트)을 그대로 적용해 확장 가능한 시나리오.

### A. 점검 범위 확장 — "자연어 한 줄로 신규 대상"
| 아이디어 | 에이전트가 하는 일 | 가치 |
|---|---|---|
| **DBMS 점검** (MySQL/PostgreSQL/MS-SQL/Oracle) | "MySQL에 익명 계정 없는지 점검" → 조회 쿼리/명령 스크립트 생성·판정 | OS 외 DB로 확장 (§2-1, 대상별 토대·개발 필요) |
| **WEB/WAS 점검** (Apache/Nginx/IIS/Tomcat) | 설정파일 조회 스크립트 자동 생성 | 설정 기반 점검에 최적 (§2-2) |
| **네트워크 장비** (스위치/라우터/방화벽) | SSH/SNMP 조회 명령 생성 — Agentless 구조 그대로 | 인프라 전체 커버 (§2-3) |
| **컨테이너/클라우드** (Docker/K8s, AWS/Azure/GCP) | CLI/API 조회 스크립트 생성 | 차별화 영역 (§2-4) |

### B. 위협 대응 속도
| 아이디어 | 에이전트가 하는 일 | 가치 |
|---|---|---|
| **신규 CVE 신속 점검** | CVE 공개 시 자연어 등록 → 영향 버전·패치 확인 스크립트 생성·판정 | 긴급 대응 시간 단축 (§3-1) |
| **고객 커스텀 기준 항목화** | 고객 보안정책 문장 → 점검 항목 자동 정의 | 고객 요청 빠른 반영 |

### C. 검증·품질 강화
| 아이디어 | 에이전트가 하는 일 | 가치 |
|---|---|---|
| **3-way 불일치 자동 재점검** | 엇갈린 항목만 재점검 스크립트 생성·재판정 | 오탐 재검토 자동화 (§4-1) |
| **멀티 LLM 합의 확장** | LSAP + Claude + … N개 교차판정 다수결 | 경계 항목 신뢰도↑ (§4-4) |
| **적응형 스크립트** | OS/배포판 차이로 명령 다를 때 대안 자동(예: `stat -c` ↔ `stat -f`) | 스크립트 깨짐 자동 보정 |

### D. 운영 자동화
| 아이디어 | 에이전트가 하는 일 | 가치 |
|---|---|---|
| **점검 → 조치 가이드/스크립트 제안** | 취약 판정 시 조치 명령까지 생성(실행은 사람 승인 게이트) | 점검~조치 루프 완성 (§5) |
| **컴플라이언스 다중 매핑** | 동일 점검을 KISA/ISMS-P/CSAP/PCI-DSS 기준으로 동시 표기 | 규정별 리포트 자동 (§5-1) |
| **정기 자동 회귀 + 보고** | 스케줄러로 신규 항목 자동 점검 → 텔레그램 보고 | 정기 자동 점검·보고 |
| **자연어 Ad-hoc 임시 점검** | "이 서버 telnet 열렸나?" 즉석 질의 → 즉시 점검 | 운영자 즉답 |

> 공통: 모든 아이디어가 **사내 LLM 전수 + 사람 게이트 2개 + 외부 전송 차단** 골격 위에서 동작한다.
> **단, 각 대상(DB/WAS/네트워크 등)은 "즉시 확장"이 아니다** — OS처럼 ① 대상별 **점검·정합성 토대** ② **환경 세팅**(접속수단·자격증명) ③ **대상별 수집 스크립트 신규 작성**(예: DB는 SQL 실행수단 추가)이 선행돼야 하는 별도 개발 과제다. (skill이 스크립트를 생성해도, 실행수단과 정합성 토대가 없으면 동작·정확판정 불가)
