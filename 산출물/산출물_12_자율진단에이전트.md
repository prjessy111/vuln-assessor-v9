# 산출물 12 — 자율 진단 에이전트 (Claude Skill 기반 자율 진단 루프)

> 신규 핵심 차별 기능. "진단 도구"를 넘어, **신규 점검 항목을 자연어로 정의하면
> 스크립트 생성 → 실행 → 판정·검증까지 사람 개입 최소로 자동 수행**하는 에이전트.
> 대상: Vuln Assessor v9 / 판정 LLM: **사내 LSAP(qwen3.5-122b-fast)** / 보고: Hermes 메신저 게이트웨이.
>
> **이 과제에서의 목적**: ① 사내 SecuMS 점검 결과의 **정합성 검증 자동화**(수작업은 정책별로, 또 SecuMS 패치마다 반복 확인해야 해 인력·시간 한계 → 자동화 필요), ② 자율 에이전트로 **신규 점검항목 요청 시 신속 대응**이 가능함을 사내 환경에서 입증 → 향후 **제품 개발에 활용**. (전 영역·전 고객 자동화가 목표가 아니라, 가능성을 보이는 선도적 PoC)

## 1. 한 줄 요약
**"공장(기존 진단 시스템)" 위에 "작업반장(에이전트)"을 얹었다.**
신규 종목이 들어와도 사람이 룰·스크립트를 일일이 짜지 않고, 자연어 설명만 주면
에이전트가 점검 스크립트와 판정을 만들고, 사람은 **두 개의 게이트(실행 승인 / 판정 확정)**만 검수한다.

## 3. 보안 설계 (사내 우선·외부 차단)
- **판정·생성 전부 사내 LSAP**(qwen, 무료·내부). raw 점검 데이터 외부 유출 없음.
- **외부 LLM(Claude)은 기본 차단** — `AGENT_ALLOW_EXTERNAL=true` 명시 없이는 코드 레벨에서 거부.
- **메신저 보고는 "판정 요약"만** 전송(raw 미포함). 폐쇄망에서는 결과 붙여넣기 방식 지원.
- **사람 확인 게이트 2개를 코드로 강제** — 승인 전 실행 불가, blocked 스크립트 승인 불가, 애매 판정은 확정 전 통과 불가.

## 5. 구성 모듈 (`src/agent/`)
| 모듈 | 역할 |
|---|---|
| `itemRegistry` | 자연어 항목 편입 + 상태머신(draft→…→confirmed) |
| `scriptGenerator` | 항목 설명 → sh/PowerShell 읽기전용 점검 스크립트 (LSAP) |
| `safetyGate` | 생성 스크립트 위험명령 정적 차단(blocked/warning/safe) |
| `scriptRunner` | 승인 스크립트 원격 실행(SSH/WinRM), 자격증명=servers.csv |
| `autoJudge` | raw → 양호/취약/점검불가 + 근거·조치·신뢰도, 저신뢰·불일치 검토 플래그 |
| `crossVerify` | 사내 재검토 → LSAP/재검토/SecuMS 3소스 합의(§4-4) |
| `ruleExport` | 확정 항목 → v2 룰 YAML(룰셋 편입) |
| `hermesNotify` | 결과 요약 메신저(Hermes) 보고 |
| `pipeline` | 단계 연결 + 사람 게이트 강제 |

## 6. 의의 / 확장
- **사람의 일이 "전수 작성"에서 "예외 검수"로 축소** — 합의된 항목은 통과, 엇갈린 것만 사람이 본다(검증 시스템의 핵심 가치).
- **신규 항목 추가 비용 절감** — 같은 OS 내 신규 CVE/가이드/고객 기준을 자연어로만 추가. ※ **현재 대상은 OS(Linux/Windows)뿐**이며, 이게 가능한 건 기존 점검·정합성을 맞춰뒀고 환경(서버·자격증명·SSH/WinRM·LLM)이 세팅돼 있어서다. (다른 대상·범위로의 확장은 §7 — 별도 과제)
- **하이브리드 검증** — 전수는 무료 사내 LSAP, 핵심·애매 항목만 선택적 2차 검증으로 정확도와 비용을 동시에.

## 7. 에이전트로 구현 가능한 아이디어 (참고 — 미확정, 방대)
> 같은 메커니즘으로 넓혀볼 수 있는 방향. **모두 검증 전 아이디어**이며 각각 별도 과제다.

- **점검 대상 확장** (DB / WEB·WAS / 네트워크 / 클라우드): 대상마다 환경 세팅·접속 테스트·정합성 토대·수집 스크립트를 **새로** 갖춰야 하는 **큰 작업**(특히 DB는 종류가 다양). 1인 단기 불가 — 인력·시간 필요.
- **신규 CVE·고객 기준 신속 항목화** (같은 OS 대상): 자연어 등록으로 빠르게 점검.
- **검증 고도화**: 3-way 불일치 자동 재점검, 멀티 LLM 합의.
- **정책 다양성(과제)**: 현재 기준은 **금융보안원(FSI) 2026 정책 하나**. 고객사마다 다른 규정(ISMS-P·CSAP·주요정보통신기반시설·PCI-DSS 등)을 쓰므로 정책별 **기준표·매핑 + 리포트 양식까지 새로** 만들어야 함.
- **운영 연계**: 정기 자동 점검·보고.
- **환경 제약(과제)**: 자동 실행(SSH/WinRM)은 사내·관리 환경에서나 가능 — 고객사는 보통 포트를 안 열어주고, 특히 **WinRM은 보안상 취약해 대부분 차단**. 폐쇄망 방식(결과 붙여넣기)·별도 접근이 필요. **(본 구현은 사내 테스트 환경 전제)**
- **정합성도 진행 중 과제**: 현재 84~88%(사내 LLM, 경계 항목은 실행별 변동) — 100% 고정이 아니라 지속 개선·관리 대상.

> 어느 방향이든 **사내 LLM·사람 게이트·외부 전송 차단** 골격을 유지한다.

> ※ 본 루프는 **사내 테스트 환경에서 동작을 실증한 초기(PoC)** 단계다. 실제 고객 환경 적용은 접속 제약(SSH/WinRM 차단)·대상 다양성·정합성 한계 등으로 쉽지 않으며, **이 난이도가 동일 형태의 상용 제품이 드문 이유일 수 있다** — 즉 어려운 문제에 대한 선도적 시도다(Claude Skill·Hermes 등 최신 기술 기반).

## 8. 한계와 개선 방향 (현실 제약 ↔ 개선 포인트)
> 장밋빛이 아니라, 실제 적용에서 부딪히는 제약과 **각각을 어떻게 풀지**를 적는다.

| 현실 제약 | 개선 포인트(방향) |
|---|---|
| **접속 제약** — 고객사는 SSH/WinRM을 잘 안 엶 (특히 WinRM은 보안상 대부분 차단) | 폐쇄망 모드 표준화(스크립트 배포 → 대상에서 수동 실행 → 결과 업로드/붙여넣기), read-only 점검 전용 계정·제한 채널 협의, 임시 수집기 배치 후 결과만 회신 |
| **대상 다양성** — OS만 됨, DB/WAS는 종류별 접속·점검 방식 상이 | `scriptRunner`에 실행 어댑터 추가(DB=SQL 커넥터, WAS=설정 조회), 대상별 점검 템플릿 라이브러리화, 수요 높은 1~2종(MySQL·Tomcat)부터 단계 확장 |
| **정책 다양성** — 현재 FSI 2026 하나, 고객마다 규정 상이 | 기준표(JSON) 외부화 구조를 살려 **정책별 기준표·매핑·리포트 템플릿**을 추가하는 형태로, 정책↔항목 매핑 테이블 운영 |
| **정합성 미완** — 84~88%, 경계 변동, 제품 패치마다 재검증 부담 | N회 다수결로 변동 흡수, **SecuMS 패치 감지 → 자동 회귀 측정(정답지 대비)** 파이프라인(CI/스케줄화), 불일치만 사람 검토 큐 |
| **LLM 속도·비결정성** — 호출당 ~30초, 경계 항목 흔들림 | 결과 캐싱·배치·동시성 조정, 1차 룰 + 2차 LLM 하이브리드로 LLM 호출 최소화 |
| **생성 스크립트 신뢰성** — LLM이 틀린 스크립트 생성 가능 | 안전게이트 + 사람 승인(적용됨), 골든 샘플 회귀 테스트, 위험 명령 화이트/블랙리스트 강화 |
