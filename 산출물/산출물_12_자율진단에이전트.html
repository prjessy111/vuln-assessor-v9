왜 핵심인가 — "공장" 위에 "작업반장"

기존 진단 시스템이 공장이라면, 자율 에이전트는 작업반장이다. 신규 종목(CVE·가이드·고객 기준)이 들어와도 사람이 룰·스크립트를 일일이 짜지 않는다. 자연어로 "이걸 점검해"라고만 하면 에이전트가 스크립트와 판정을 만들고, 사람은 예외(불일치·애매)만 검수한다. → 검증 시스템의 가치(전수 자동 + 예외만 사람)를 그대로 구현.

1. 파이프라인 — 5단계 + 사람 게이트 2개

① 자연어 항목 편입→ ② 스크립트 자동 생성
+ 안전게이트(위험명령 차단)→ 🚦 게이트1
실행 승인→ ③ 대상 실행·수집
SSH / WinRM→ ④ raw 자동 판정
근거·조치·신뢰도→ ⚖ 교차검증

사람 확인 게이트는 코드로 강제 — 승인 전 실행 불가, 위험(blocked) 스크립트 승인 불가, 애매 판정은 확정 전 통과 불가.

6. 의의 / 확장

  • 사람의 일이 "전수 작성"에서 "예외 검수"로 축소 — 합의된 항목은 통과, 엇갈린 것만 사람이 본다.
  • 신규 항목 추가 비용 절감 — 같은 OS 내 신규 CVE/가이드/고객 기준을 자연어로만 추가. 현재 대상은 OS(Linux/Windows)뿐이며, 기존 점검·정합성과 환경(접속·자격증명)이 갖춰져 있어 가능. (다른 대상·범위 확장은 §7 — 별도 과제)
  • 하이브리드 검증 — 전수는 무료 사내 LSAP, 핵심·애매 항목만 선택적 2차 검증으로 정확도·비용 동시 확보.

※ 본 기능은 핵심 차별 요소로, 시간 관계상 메인 발표에서 빠졌던 부분을 보강 제출한다.

7. 에이전트로 구현 가능한 아이디어 (참고 — 미확정, 방대)

같은 메커니즘으로 넓혀볼 수 있는 방향. 모두 검증 전 아이디어이며 각각 별도 과제다.

  • 점검 대상 확장 (DB / WEB·WAS / 네트워크 / 클라우드) — 대상마다 환경 세팅·접속 테스트·정합성 토대·수집 스크립트를 새로 갖춰야 하는 큰 작업(특히 DB는 종류가 다양). 1인 단기 불가 — 인력·시간 필요.
  • 신규 CVE·고객 기준 신속 항목화 (같은 OS 대상)
  • 검증 고도화: 3-way 불일치 자동 재점검, 멀티 LLM 합의
  • 운영 연계: 컴플라이언스 다중 매핑, 정기 자동 점검·보고
  • 환경 제약(과제): 자동 실행(SSH/WinRM)은 사내·관리 환경에서나 가능 — 고객사는 보통 포트를 안 열어주고, 특히 WinRM은 보안상 취약해 대부분 차단. 폐쇄망 방식(결과 붙여넣기)·별도 접근이 필요. (본 구현은 사내 테스트 환경 전제)
  • 정합성도 진행 중 과제: 현재 84~88%(사내 LLM, 경계 항목 변동) — 100% 고정이 아니라 지속 개선·관리 대상.

어느 방향이든 사내 LLM·사람 게이트·외부 전송 차단 골격을 유지한다.

※ 본 루프는 사내 테스트 환경에서 동작을 실증한 초기(PoC) 단계다. 실제 고객 환경 적용은 접속 제약(SSH/WinRM 차단)·대상 다양성·정합성 한계 등으로 쉽지 않으며, 이 난이도가 동일 형태의 상용 제품이 드문 이유일 수 있다 — 어려운 문제에 대한 선도적 시도(Claude Skill·Hermes 등 최신 기술 기반).
