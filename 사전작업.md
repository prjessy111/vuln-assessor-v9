# 사전작업 — Windows 대상 서버 WinRM 원격 진단 준비

> vuln-assessor 가 Windows 서버를 원격 진단하려면 **WinRM** 접속이 열려 있어야 한다.
> 이 문서는 **앱이 도는 PC(컨트롤러)** 에서만 설정해 대상 서버에 접속을 여는 절차다.
> (대상 서버에 WinRM 이 이미 켜져 있는 경우 — 컨트롤러 설정만으로 완료된다.)

---

## 0. 개념 정리

| 구분 | 역할 | 이 문서에서 건드리는가 |
|---|---|---|
| **컨트롤러 PC** | vuln-assessor(node) 앱이 도는 PC. 접속을 *거는* 쪽 | ✅ 여기서만 작업 |
| **대상 서버** | 진단 대상 Windows 서버 (예: jessy107 / 192.168.159.107) | ❌ 손대지 않음 (WinRM 5985 만 열려 있으면 됨) |

- 접속 방식: **WinRM / HTTP 5985**
- 인증: 컨트롤러의 `TrustedHosts` 에 대상 IP 등록 필요
- `TrustedHosts` 는 **접속을 거는 쪽(컨트롤러)** 에 설정해야 효과가 있다.

---

## 1. 관리자 권한 PowerShell 열기

- **Win + X** → **Windows PowerShell(관리자)** / **터미널(관리자)**
- 또는 시작 → `powershell` → 우클릭 → **관리자 권한으로 실행**
- 창 제목이 **"관리자: Windows PowerShell"**, 프롬프트가 `C:\WINDOWS\system32>` 인지 확인

> ⚠️ 일반 창에서 실행하면 `Access is denied` / `WinRM 서비스를 열 수 없습니다` 오류가 난다. 반드시 관리자 창에서.

---

## 2. 컨트롤러의 WinRM 클라이언트 서비스 켜기

```powershell
Start-Service WinRM
Set-Service WinRM -StartupType Automatic
```

- 메시지 없이 넘어가거나 "이미 실행 중" → 정상

---

## 3. 대상 서버를 신뢰 대상(TrustedHosts)에 등록

```powershell
Set-Item WSMan:\localhost\Client\TrustedHosts -Value "192.168.159.107" -Concatenate -Force
Restart-Service WinRM
```

- 여러 대면 쉼표로: `-Value "192.168.159.167,192.168.159.107"`
- `-Concatenate` 덕분에 기존 등록은 유지하고 추가된다.
- `Restart-Service` 시 "중지 대기 중" 경고가 한두 번 떠도 정상. 10초쯤 기다리면 끝남.

**등록 확인:**
```powershell
Get-Item WSMan:\localhost\Client\TrustedHosts
```
→ `Value` 칸에 대상 IP 들이 보이면 OK.

---

## 4. 네트워크 연결 확인

```powershell
Test-NetConnection 192.168.159.107 -Port 5985
```

- **`TcpTestSucceeded : True`** → 5단계로
- **`False`** → 대상 서버의 WinRM 이 꺼져 있거나 방화벽 차단 (→ 아래 "대상 서버가 막힌 경우" 참고)

---

## 5. 실제 접속 테스트

```powershell
$cred = Get-Credential
```
팝업 입력:
- 사용자 이름: `Administrator`
- 암호: `omni007!`  *(servers.csv 의 해당 서버 비밀번호)*

```powershell
Invoke-Command -ComputerName 192.168.159.107 -Credential $cred -ScriptBlock { hostname; whoami }
```

- 대상의 **hostname + `대상\administrator`** 가 출력되면 → **성공** 🎉
- hostname 만 떠도 인증·실행이 통과한 것 (= 성공)
- 빨간 에러면 → 아래 "자주 나는 에러" 참고

---

## 6. 마무리

1. **vuln-assessor(node) 앱 재시작** — 새 WinRM 설정 반영
2. 앱의 **서버 관리 / 진단** 화면에서 대상 서버 수집·진단 실행
3. 스크립트 배포·실행 → 결과 XML 수신 확인

## 부록 A — 자주 나는 에러

| 에러 | 원인 | 조치 |
|---|---|---|
| `Access is denied` (서비스 제어 시) | PowerShell 이 관리자 아님 | 관리자 창으로 다시 열기 (1단계) |
| `WinRM client cannot process the request` / `신뢰` | TrustedHosts 미등록 | 3단계 재확인 |
| `Access denied` (Invoke-Command 시) | 계정/비번 오류 또는 대상 인증정책 | 비번 확인, 대상 Basic 인증(부록 B) |
| 연결 timeout / `TcpTestSucceeded: False` | 대상 WinRM 미기동 또는 방화벽 5985 차단 | 부록 B |

---

## 부록 B — 대상 서버가 막힌 경우 (4단계가 False)

대상 서버에 직접 로그인(RDP) 없이, **컨트롤러에서 SMB(445)로 원격으로 WinRM 을 켜는** 방법.
[Sysinternals PsExec](https://learn.microsoft.com/sysinternals/downloads/psexec) 필요.

```powershell
psexec \\192.168.159.107 -u Administrator -p omni007! -h `
  powershell -Command "Enable-PSRemoting -Force; winrm set winrm/config/service/auth '@{Basic=\"true\"}'; winrm set winrm/config/service '@{AllowUnencrypted=\"true\"}'"
```

- 대상에 **admin SMB 접근(445)** 이 열려 있어야 동작.
- SMB 도 막혀 있으면 → 대상 서버에 한 번은 직접 접속해 WinRM 을 켜야 한다.

> ⚠️ `Basic` + `AllowUnencrypted` 는 **사내 폐쇄망 PoC 기준**. 외부망이면 HTTPS(5986)+Negotiate 로 가야 한다.

---

## 참고 — 접속 경로 (코드 기준)

- **에이전트 / 스케줄러** (`src/agent/scriptRunner.js`, `src/services/scheduler.js`): `nodejs-winrm` → **Basic 인증 / HTTP 5985**
- **배포 서비스** (`src/services/scriptDeployService.js`): 기본 `powershell-only`(Invoke-Command) → 컨트롤러 **TrustedHosts** 필요

> 단순 `Invoke-Command` 테스트(5단계)는 통과해도, 앱이 `nodejs-winrm` 경로를 타면 대상에 `Basic`/`AllowUnencrypted` 가 필요할 수 있다. 그 경우 부록 B 로 처리.
