0. 가장 중요한 차이 — "계층(layer)"이 다름

두 docx가 진단한 것은 애플리케이션에 내장된 라이브러리(컴파일/번들된 OSS·JAR)이고, 우리 스캔은 OS에 설치된 프로그램(레지스트리 Uninstall)이다. 서로 다른 깊이라 대부분 겹치지 않는다.
  • 107 docx = 제품 내장 네이티브 OSS: OpenSSL 1.0.2o, zlib 1.2.3, POCO 1.4.2, libexpat 2.0.1, PCRE 8.x, SQLite 3.7.7.1, TinyXML 2.6.2, SIGAR 1.6.4, PostgreSQL 13.0
  • 147 docx = Java 앱 의존성(JAR): Apache Commons(BeanUtils·Collections·Compress·Text 등), Log4j, PDFBox, POI, Tika, CXF, mime4j, PostgreSQL JDBC 42.3.1, PostgreSQL 14.4
  • 우리 = 레지스트리 설치 프로그램 (Flash, Java 런타임, Edge, SQL Server, PostgreSQL 등)
→ 내장 .dll/.jar 은 레지스트리에 안 나오므로 우리 현재 스캔으론 docx 항목 대부분이 안 보인다.

3. 정합성 — 실측 (OSS 라이브러리 수집 후, 2026-06-30)

재수집(JAR·네이티브·strings)으로 OSS 계층까지 탐지 → docx와 "틀린" 게 아니라 "겹치고 + 더 많이 + 일부 누락":
  • 일치(독립 검증): OpenSSL 1.0.2o · zlib 1.2.3 · SQLite · PostgreSQL → docx와 동일하게 우리도 탐지 ✅
  • 초과 탐지(docx에 없음): log4j · commons-collections · hadoop · jetty · struts 등 — 우리가 더 잡음
  • 누락: POCO 1.4.2 · TinyXML 2.6.2 · SIGAR 1.6.4 — CVE DB·strings 프로브 미커버 (추가하면 탐지 가능)
리포트 내 고유 CVE: 107 = 95건, 147 = 59건. 모두 판정(유력/검토필요/패치됨) + 표기 근거 포함.

4. ★ 우리가 고칠 것 (action items)

① 내장 라이브러리 스캔 부재 (가장 큰 갭)

docx 항목 대부분(OpenSSL/zlib/Apache Commons/Log4j 등)은 제품에 번들된 .dll/.jar라 레지스트리에 안 보인다. 이를 잡으려면:
  • 파일시스템 스캔: 설치 경로의 .dll 버전 문자열, .jar manifest(예: META-INF/MANIFEST.MF, 파일명 log4j-core-2.x.jar) 수집
  • 또는 SBOM 수집(CycloneDX/SPDX) → 의존성 인벤토리 매칭
  • → 이걸 추가하면 docx와 같은 층을 커버 (현재는 OS 설치 프로그램만)

② 버전 정밀도

  • PostgreSQL "13"처럼 minor 누락 → 정밀 버전(13.7 등) 수집·표기 보강 (패치 여부 판정 정확화)
  • 147 PostgreSQL 13 vs docx 14.4 불일치 실측 확인

③ CVE DB 확장

  • Apache Commons/Log4j/PDFBox/JDBC 등 Java 생태계 CVE를 NVD 동기화로 확보 (현재 베이스라인 27건)
  • KEV 제품 탐지는 버전 미검증 자문 → NVD CPE 연동으로 버전 정밀 매칭 전환

5. 결론 (포지셔닝)

  • docx = 한 제품의 내장 라이브러리를 수기로 깊게 분석(시점 고정·참고). OS·설치프로그램·KEV·자동갱신은 없음.
  • 우리 = OS 패치 + 설치 프로그램 + KEV + NVD 자동갱신을 여러 서버에 자동으로. SecuMS가 못 하는 CVE 강점.
  • 상호보완: docx의 "내장 라이브러리 심층"을 우리가 흡수하려면 ④①(파일시스템/SBOM 스캔)이 다음 과제.
  • 이미 입증된 것: PostgreSQL을 docx와 독립적으로 동일 탐지 = 우리 정확성. 나머지 미탐은 "범위(계층)" 차이지 오류가 아님.
